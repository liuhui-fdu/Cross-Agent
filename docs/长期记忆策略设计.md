# Cross-Agent 长期记忆策略设计

本文档说明 Cross-Agent 当前混合长期记忆架构。系统以规则作为不可绕过的控制面，以可选 LLM 作为语义候选层，重点覆盖抽取、更新、遗忘、检索、使用、安全与拒答七个环节，并披露关键阈值、状态迁移和计算公式。

当前实现主线是：

```text
用户会话
  -> 硬安全预扫描与用户指令识别
  -> 规则候选 + 可选 LLM 语义候选
  -> 候选合并与去重
  -> Schema / 边界 / 置信度校验
  -> 隐私准入与写入治理
  -> memory_state / memory_events
  -> 查询规划
  -> 混合检索与重排
  -> 隐私披露过滤
  -> EvidenceBundle
  -> 受证据约束的回答
  -> ResponseGuard 校验
```

核心原则是：**模型只能提出候选，规则拥有最终决定权；抽取不等于记住，检索不等于使用，没有证据不编造。**

### 混合架构职责边界

| 层 | 负责 | 不负责 |
|---|---|---|
| 规则抽取层 | 已知高精度槽位、敏感预扫描、用户不保存/忘记指令 | 开放领域的完整语义理解 |
| LLM 语义层 | 否定、修正、时间范围、开放事实与偏好候选 | 最终写入、权限、敏感准入 |
| WritePolicy | Schema、身份边界、数值范围、payload 大小、推断阈值 | 生成新事实 |
| PrivacyPolicy | 禁存模式、敏感存储许可 | 判断业务偏好是否正确 |
| CandidateResolver | 两路候选统一过滤、合并、评分、槽位冲突消解和 Top 10 | 修改数据库状态 |
| Governor | create / reinforce / supersede / archive / reject | 自由生成候选内容 |
| Store | 原子状态迁移、脱敏审计 | 绕过 Governor 自行判断 |

## 1. 数据模型

### 1.1 记忆类型

`MemoryType` 定义长期记忆的类型：

| 类型 | 含义 | 示例 |
|---|---|---|
| `fact` | 相对稳定的事实 | 用户叫林澈，住在杭州滨江 |
| `preference` | 偏好 | 接口示例优先用 Python，默认喝茶 |
| `task` | 待办或任务状态 | 周五前整理 Cross-Agent 架构演示材料 |
| `relation` | 关系类信息 | 由 LLM 语义候选层抽取并接受统一治理 |
| `event` | 会话证据或历史事件 | 一轮原始会话的摘要、transcript、keywords |
| `summary` | 汇总型记忆 | 由 LLM 语义候选层抽取并接受统一治理 |
| `sensitive` | 敏感记忆 | 默认拒绝存储，只有配置显式允许时才可准入 |

代码位置：`src/cross_agent/models.py`

### 1.2 记忆状态

`MemoryStatus` 定义记忆的生命周期状态：

| 状态 | 含义 | 当前使用情况 |
|---|---|---|
| `active` | 当前可用 | 检索默认只召回 active |
| `tentative` | 暂定状态 | 同时间、无时间或低强度推断冲突，暂不影响 active |
| `superseded` | 已被新值替代或保存为历史 | Java -> Python；迟到旧值作为 historical 保存 |
| `archived` | 归档 | 用户明确“忘记”后进入该状态，不再参与 active 检索 |
| `rejected` | 拒绝 | 拒绝事件记录在 `memory_events`，不进入 `memory_state` |

### 1.3 两张核心表

`memory_state` 保存当前和历史状态：

| 字段 | 用途 |
|---|---|
| `memory_id` | 记忆 ID |
| `tenant_id`, `user_id` | 多租户和用户隔离 |
| `type`, `subject`, `predicate`, `scope` | 类型化槽位 |
| `value_json` | 结构化值 |
| `assertion_mode` | 断言方式，例如 explicit |
| `literalness` | 字面性，例如 literal / uncertain |
| `confidence` | 置信度 |
| `importance` | 重要性 |
| `sensitivity` | 敏感级别 |
| `status` | active / superseded 等 |
| `source_turn_ids_json` | 来源 turn |
| `source_session_id` | 来源 session |
| `valid_from`, `valid_to` | 有效时间 |
| `supersedes` | 新记录替代的旧记录 |
| `extraction_source` | rule / llm / hybrid |
| `write_score` | 候选进入状态时的统一写入分数 |

`memory_events` 保存操作审计：

| 字段 | 用途 |
|---|---|
| `event_id` | 事件 ID |
| `memory_id` | 被创建、强化或替换的记忆 ID |
| `operation` | create / create_historical / create_tentative / promote_tentative / reinforce / supersede / archive / reject |
| `payload_json` | 操作原因和目标记忆；reject 默认只保留脱敏元数据 |
| `source_turn_ids_json` | 来源 turn |
| `event_time` | 业务发生时间 |
| `system_time` | 系统写入时间 |
| `actor` | 当前为 system |

代码位置：`src/cross_agent/store/sqlite_store.py`

## 2. 抽取策略

抽取层输出的是 `MemoryCandidate`，而不是直接写入长期记忆。每条候选记忆必须携带：

```text
tenant_id
user_id
memory_type
subject
predicate
value
scope
assertion_mode
literalness
confidence
importance
sensitivity
source_turn_ids
source_session_id
valid_from
valid_to
```

这样后续治理层才能判断它是否可信、是否重复、是否覆盖旧值、是否敏感、是否可披露。

### 2.1 Session evidence 抽取

当前默认 `session_evidence_mode=useful`，不再把每个非空会话无差别写入长期记忆。满足以下任一条件才生成 `event/session_evidence`：

- 已抽取到结构化 fact / preference / task。
- 命中“记住、偏好、需要、计划、以后、目前、改成、住在、我叫”等持久性信号。
- 调用方通过 `metadata.force_session_evidence=true` 显式要求保留，例如 ActMem 检索评测。

纯查询如“今天天气怎么样？”以及“根据之前聊过的内容告诉我”“我上次住哪家酒店”“帮我按以前的习惯推荐”等检索型请求，都会在规则抽取和 LLM 抽取之前识别，不生成长期事件，也不调用语义写入模型。只有以“帮我/请帮我”发起的当前命令才命中该规则，句中“以后可能会让你帮我”仍按声明处理。更新、纠正、完成任务和遗忘指令不属于纯查询，仍进入写入链路。配置也支持：

```text
session_evidence_mode = all     # 所有非空会话
session_evidence_mode = useful  # 默认，仅未来可能有用的会话
session_evidence_mode = none    # 不保存原始会话证据
```

生成的证据结构为：

```json
{
  "type": "event",
  "predicate": "session_evidence",
  "scope": "actmem_session",
  "value": {
    "session_id": "demo_cn_03",
    "summary": "user: 我喜欢喝咖啡，尤其是早上做设计评审前。",
    "transcript": "user: 我喜欢喝咖啡，尤其是早上做设计评审前。",
    "keywords": ["user", "我喜欢喝咖啡", "咖啡", "设计", "评审前"],
    "metadata": {
      "demo_round": 3,
      "purpose": "抽取 preferred_drink=咖啡。"
    }
  }
}
```

用途：

- 保留原始证据，便于回答时追溯来源。
- ActMem 风格评测中可以基于原始会话证据召回。
- 当结构化抽取不确定时，仍可保留低置信证据。

### 2.2 结构化抽取

当用户表达是字面、明确、非禁存时，会额外抽取结构化槽位。

带有“不要公开、别在公开总结中提、只私下、confidential”等披露范围限制的第三方内容按 `sensitive` 处理：原文不会发送给语义抽取模型，也不会进入 `memory_state`，审计事件只保留脱敏后的拒绝原因。

示例 1：身份事实

```text
用户：你好，我叫林澈，现在住在杭州滨江。
```

抽取：

```json
[
  {
    "type": "fact",
    "predicate": "name",
    "scope": "identity",
    "value": {"text": "林澈"},
    "confidence": 0.96,
    "importance": 0.82
  },
  {
    "type": "fact",
    "predicate": "current_residence",
    "scope": "current_residence",
    "value": {"text": "杭州滨江"},
    "confidence": 0.90,
    "importance": 0.76
  }
]
```

示例 2：偏好

```text
用户：团队技术栈改了，我现在主要用 Python，之后接口示例优先用 Python。
```

抽取：

```json
{
  "type": "preference",
  "predicate": "preferred_programming_language",
  "scope": "coding_examples",
  "value": {"text": "python"},
  "confidence": 0.88,
  "importance": 0.74
}
```

示例 3：任务

```text
用户：记住我需要周五前整理 Cross-Agent 的架构演示材料。
```

抽取：

```json
{
  "type": "task",
  "predicate": "user_task",
  "scope": "task:2902041d3d",
  "value": {
    "text": "周五前整理 Cross-Agent 的架构演示材料",
    "state": "open"
  },
  "confidence": 0.78,
  "importance": 0.70
}
```

### 2.3 字面性策略

抽取器会检测非字面表达。当前规则：

```text
如果 transcript 包含 haha / just kidding / 哈哈 / 开玩笑 / 玩笑
  literalness = uncertain
  confidence = 0.45
否则
  literalness = literal
  confidence = 0.80
```

示例：

```text
用户：哈哈我最喜欢凌晨三点改需求了，开玩笑的。
```

结果：

- 保存 `event/session_evidence`
- `literalness=uncertain`
- `confidence=0.45`
- 不抽取 `preferred_work_time=凌晨三点`

这避免把玩笑、反话、假设句写成长期画像。

### 2.4 重要性计算

默认重要性来自配置：

```json
{
  "default_importance": 0.55
}
```

当前启发式公式：

```text
importance = min(default_importance + 0.04 * signal_count, 0.95)
```

其中 `signal_count` 是以下信号词命中的数量：

```text
remember, note, log, important, requires, depends, status, updated, added,
记住, 重要, 需要, 更新, 改成
```

示例：

```text
用户：记住我需要周五前整理 Cross-Agent 的架构演示材料。
```

命中 `记住`、`需要`，所以 session evidence 的重要性会高于默认值：

```text
importance = min(0.55 + 0.04 * 2, 0.95) = 0.63
```

结构化任务本身使用更高的固定重要性，例如 `0.70`。

### 2.5 写入准入规则

规则候选和 LLM 候选必须通过同一套写入策略：

```text
tenant_id 和 user_id 非空
candidate.subject == candidate.user_id
predicate / scope 符合槽位字符和长度约束
value 是非空对象
JSON payload <= 8000 字符
source_session_id 和 source_turn_ids 非空
sensitivity 属于 low / medium / high / sensitive / forbidden
0 <= confidence <= 1
0 <= importance <= 1
candidate.confidence >= writer.min_confidence
如果 assertion_mode == inferred：
    confidence >= writer.min_inferred_confidence
candidate.literalness in writer.literalness_allowlist
```

默认配置：

```json
{
  "min_confidence": 0.35,
  "min_inferred_confidence": 0.72,
  "max_candidate_chars": 8000,
  "max_candidates_per_session": 24,
  "final_candidate_top_k": 10,
  "candidate_conflict_margin": 0.08,
  "literalness_allowlist": ["literal", "quotation", "uncertain"]
}
```

注意：`uncertain` 当前允许进入 session evidence，是为了保留可审计证据；但结构化抽取层会阻止 uncertain 进入高置信结构化槽位。

### 2.6 LLM 语义候选层

当同时满足以下条件时启用语义抽取：

```text
writer.semantic_extraction_enabled == true
llm.enabled == true
API key 可用
会话未命中 forbidden
会话未包含 do_not_store 指令
```

LLM 只返回 JSON 候选，不直接访问 Store。候选格式：

```json
{
  "candidates": [
    {
      "action": "store",
      "type": "preference",
      "predicate": "preferred_work_location",
      "scope": "deep_work",
      "value": {"text": "安静的办公室"},
      "assertion_mode": "explicit",
      "literalness": "literal",
      "confidence": 0.88,
      "importance": 0.64,
      "sensitivity": "low",
      "valid_from": "2026-06-29",
      "valid_to": null,
      "source_turn_ids": ["turn_1"]
    }
  ]
}
```

语义层允许输出 `fact / preference / task / relation / summary`，不允许生成 `event/session_evidence` 或 `sensitive` 类型。模型输出还会被重新绑定到当前 session 的 `tenant_id/user_id/source_turn_ids`，不能自行指定其他用户。

### 2.7 统一候选池与故障回退

规则和 LLM 不再按“规则优先、模型补位”的顺序截取。两路候选先进入同一个原始候选池：

```text
rule_candidates[:24] + llm_candidates[:24]
```

这里的 24 是每个来源的防失控上限，不是最终写入数量。随后所有候选统一交给 `CandidateResolver`。LLM 超时、返回非法 JSON或字段异常时，`HybridMemoryExtractor` 记录错误并自动退回规则候选，不中断整轮写入。

### 2.8 统一过滤与重复合并

重排前先执行不可绕过的硬过滤：

```text
PrivacyPolicy.can_store(candidate)
WritePolicy.admits(candidate)
```

过滤失败的候选生成脱敏 reject 审计，不参与排序。通过候选按以下键合并：

```text
type + subject + predicate + scope + normalized(sorted JSON value)
```

相同值由规则和 LLM 同时抽到时：

```text
extraction_source = hybrid
confidence = min(0.99, max(confidence) + 0.05)
importance = max(importance)
source_turn_ids = union(source_turn_ids)
assertion_mode 优先 explicit
literalness 优先 literal
```

因此跨来源一致性表现为置信度增强，而不是简单保留先到候选。

### 2.9 写入评分与统一重排

除用户控制指令外，所有规则和 LLM 候选使用同一个评分公式：

```text
write_score =
    0.30 * confidence
  + 0.20 * importance
  + 0.15 * source_reliability
  + 0.20 * temporal_score
  + 0.10 * durability
  + 0.05 * specificity
  + explicit_bonus
  + correction_bonus
  - uncertainty_penalty
  - sensitivity_penalty
```

时间分数使用按类型半衰期衰减：

```text
age_days = max(0, now - valid_from)
temporal_score = exp(-ln(2) * age_days / half_life_days)
```

没有可解析 `valid_from` 时，时间分数为 `0.60`。当前半衰期：

| 类型 | 半衰期 |
|---|---:|
| fact | 730 天 |
| relation | 365 天 |
| summary | 365 天 |
| preference | 180 天 |
| event | 90 天 |
| task | 30 天 |
| sensitive | 30 天 |

`valid_from` 超过当前系统时间 1 天以上会在硬过滤阶段拒绝，原因是 `valid_from_too_far_in_future`。

来源可靠度：

```text
rule                  = 1.00
hybrid                = 1.00
llm + explicit        = 0.85
llm + inferred        = 0.65
unknown               = 0.60
```

类型持久度：

```text
fact       = 0.90
task       = 0.86
preference = 0.82
relation   = 0.75
summary    = 0.60
event      = 0.40
sensitive  = 0.20
```

奖励与惩罚：

```text
explicit_bonus       = 0.12
correction_bonus     = 0.15  # 命中“改成、不再、更新为、instead”等修正信号
uncertainty_penalty  = 0.25  # literalness=uncertain
medium penalty       = 0.08
high/sensitive       = 0.30
forbidden            = 1.00  # 实际会在硬过滤阶段拒绝
```

`do_not_store` 和 `forget` 属于用户控制指令，固定：

```text
write_score = 10.0
```

保证用户控制不会被普通候选挤出 Top 10。

### 2.10 槽位冲突消解

评分后按以下槽位分组：

```text
type + subject + predicate + scope
```

每个槽位最多向 Governor 提交一个候选：

```text
同值候选              -> 前一步已经合并
不同值且分差 >= 0.08  -> 选择 write_score 最高者
不同值且分差 < 0.08   -> dropped_ambiguous，整个槽位本轮不更新
explicit 对 inferred   -> explicit 可直接胜出，不视为近分歧义
```

这避免同一轮先写“咖啡”再被“茶”覆盖，也避免最终值由遍历顺序决定。

### 2.11 全局 Top 10

各槽位胜出候选统一按 `write_score` 降序排列，来源只作为同分稳定排序项：

```text
所有槽位 winner
  -> sort(write_score DESC)
  -> final_candidate_top_k = 10
  -> Governor
```

规则和 LLM 没有独立写入配额。高质量 LLM 候选可以排在低价值规则事件之前，高质量规则候选也不会因为 LLM 返回数量多而被挤掉。

Resolver trace 记录：

```text
raw_candidate_count
admitted_candidate_count
merged_candidate_count
rejected_candidate_count
conflict_dropped_count
top_k_dropped_count
ranking
selected
conflicts
operations
```

### 2.12 用户控制指令

规则层优先识别：

```text
不要保存这次内容
不要把以下内容记入长期记忆
只在本次对话使用
```

这类会话转换为 `do_not_store` 控制候选，最终执行 `REJECT`，不会保存 transcript，也不会调用 LLM。

对于明确忘记指令：

```text
请忘记我的饮品偏好
删除我的居住地
清除代码语言偏好
```

系统生成 `assertion_mode=forget` 的目标槽位候选，由 Governor 执行 `ARCHIVE`。

代码位置：

- `src/cross_agent/writer/extractor.py`
- `src/cross_agent/writer/resolver.py`
- `src/cross_agent/policies/write_policy.py`

## 3. 更新策略

候选先由 `CandidateResolver` 在本轮内部统一消歧和选出 Top 10，再由 `MemoryGovernor` 根据数据库中已有状态决定创建、强化、替换或归档。前者解决“本轮候选之间谁胜出”，后者解决“胜出候选与历史状态如何交互”。

### 3.1 槽位定义

结构化记忆的槽位键：

```text
slot_key =
  tenant_id
  + user_id
  + memory_type
  + subject
  + predicate
  + scope
```

例如：

```text
demo-cn / lin-che / preference / lin-che / preferred_programming_language / coding_examples
```

这个槽位表示“林澈在代码示例语言上的当前偏好”。

### 3.2 操作决策

治理规则：

```text
if assertion_mode == do_not_store:
    operation = reject(user_requested_no_store)
elif privacy_policy.can_store(candidate) is false:
    operation = reject
elif write_policy.admits(candidate) is false:
    operation = reject
elif assertion_mode == forget:
    if active target slot exists:
        operation = archive(user_requested_forget)
    else:
        operation = reject(forget_target_not_found)
elif candidate is event/session_evidence:
    if same source_session 的 event/session_evidence exists:
        operation = reinforce
    else:
        operation = create
else:
    old = find_active_by_slot(slot_key)
    if old is None:
        operation = create
    else:
        time_order = compare(candidate.valid_from, old.valid_from)

        if same_value(old.value, candidate.value):
            if time_order == older:
                operation = reject(stale_duplicate_evidence)
            else:
                operation = reinforce

        elif time_order == older:
            operation = create_historical

        elif time_order == newer:
            if candidate is inferred and score insufficient:
                operation = create_tentative
            else:
                operation = supersede

        elif time_order == equal:
            if candidate is explicit and score >= old.score + 0.08:
                operation = supersede
            else:
                operation = create_tentative

        else:  # 任一方缺少可比较时间
            operation = score_and_assertion_based_supersede_or_tentative
```

用户不保存指令在最前；任何模型候选都必须先经过隐私和 WritePolicy；forget 只能归档当前用户当前槽位的 active 记录。`find_active_by_source_session` 只匹配 `event/session_evidence`，不会因结构化候选重排到前面而误判。

### 3.3 时间解析与硬时间门

系统支持 ISO 日期、`YYYY-MM-DD` 和 ActMem 的 `2024/02/20 (Tue) 05:25` 格式，比较前统一转换为 UTC 无时区 `datetime`。

```text
candidate.valid_from < active.valid_from  -> older
candidate.valid_from = active.valid_from  -> equal
candidate.valid_from > active.valid_from  -> newer
任一方无法解析或缺失                      -> unknown
```

时间首先是状态更新的硬门，而不只是评分项：旧候选无论分数多高都不能覆盖更新的 active；较新的显式修正可以替换旧值；无法确定时间的冲突优先进入 tentative。

### 3.4 相同值判断

当前实现将两个 value 转成排序后的 JSON，并做 normalize：

```text
same_value(left, right) =
  normalize(json.dumps(left, sort_keys=True))
  ==
  normalize(json.dumps(right, sort_keys=True))
```

示例：

```json
{"text": "python"}
```

与：

```json
{"text": "Python"}
```

在当前实现中不一定完全等价，因为结构化抽取已提前将语言统一成小写。也就是说，归一化主要由抽取器和 `normalize` 共同完成。

### 3.5 强化策略

如果同一槽位、同一值再次出现，且候选时间不早于 active，执行 `reinforce`：

```text
confidence_new = min(0.99, confidence_old + 0.02)
importance_new = min(0.99, importance_old + 0.02)
write_score_new = max(write_score_old, write_score_candidate)
extraction_source = 必要时合并为 hybrid
updated_at = now
```

示例：

```text
用户多次明确说：接口示例优先用 Python。
```

系统不会创建多条重复 Python 偏好，而是提高现有记录的可信度和重要性。

如果同值证据的时间早于 active，执行 `REJECT(stale_duplicate_evidence)`，避免重放旧会话反复强化当前状态。

### 3.6 替换、历史与暂定策略

如果同一槽位出现新值，先按时间和分数分流。

#### 较新的明确值

较新的 explicit 候选执行 `supersede`：

```text
old.status = superseded
old.valid_to = candidate.valid_from or now
old.updated_at = now

new.status = active
new.supersedes = old.memory_id
```

较新的 inferred 候选只有在：

```text
candidate.write_score >= active.write_score + 0.08
```

时才能替换；否则创建 `tentative`，不影响 active。

#### 迟到旧值

候选早于 active 且值不同时执行 `create_historical`：

```text
historical.status = superseded
historical.valid_from = candidate.valid_from
historical.valid_to = active.valid_from
active 保持不变
```

#### 同时间冲突

同时间不同值时，只有 explicit 且分数至少高出 `0.08` 才能替换，否则执行：

```text
create_tentative
status = tentative
active 保持不变
```

#### 时间缺失

```text
新候选有时间、active 无时间、且新候选 explicit -> supersede
新候选有时间但只是 inferred                  -> tentative
新候选无时间、active 有时间                   -> tentative
双方均无时间，explicit 且分数高出 0.08        -> supersede
其他情况                                      -> tentative
```

#### Tentative 确认与转正

后续候选与某条 tentative 同槽位、同值且时间不早于 tentative 时：

```text
同一 source_session 重复提交 -> reject，防止单一来源自我确认
新的 explicit 证据           -> promote_tentative
不同 Session 的足够强证据    -> promote_tentative
不同 Session 的弱证据        -> reinforce tentative
```

转正操作会原子完成：

```text
旧 active.status = superseded
tentative.status = active
tentative.supersedes = 旧 active.memory_id
confidence = min(0.99, max(原 tentative, 新证据) + 0.05)
importance = min(0.99, max(原 tentative, 新证据) + 0.02)
来源与 source_turn_ids 合并
同槽位其他 tentative 自动 archived
```

正常 `supersede` 产生确定新值时，同槽位遗留 tentative 也会自动归档，确保一个槽位只有一个 current active。

示例：Java -> Python

旧记录：

```json
{
  "type": "preference",
  "predicate": "preferred_programming_language",
  "scope": "coding_examples",
  "value": {"text": "java"},
  "status": "superseded",
  "valid_from": "2026-06-02",
  "valid_to": "2026-06-06"
}
```

新记录：

```json
{
  "type": "preference",
  "predicate": "preferred_programming_language",
  "scope": "coding_examples",
  "value": {"text": "python"},
  "status": "active",
  "valid_from": "2026-06-06",
  "supersedes": "mem_71a4d15796b14823"
}
```

示例：咖啡 -> 茶

```text
用户：最近胃不舒服，我现在改喝茶，别再默认推荐咖啡。
```

结果：

- `preferred_drink=咖啡` 变成 `superseded`
- `valid_to=2026-06-08`
- 新建 `preferred_drink=茶`
- 新值为 `active`

代码位置：

- `src/cross_agent/governor/governor.py`
- `src/cross_agent/store/sqlite_store.py`

## 4. 遗忘策略

当前实现同时支持值更新造成的软失效和用户明确要求的归档遗忘；自动衰减与物理删除仍是后续能力。

### 4.1 已实现：状态失效

当新值替代旧值时，旧记忆不会物理删除，而是：

```text
status = superseded
valid_to = 新值 valid_from
```

默认检索只读取：

```text
status = active
```

因此旧值不会进入普通回答。

### 4.2 已实现：用户主动忘记

例如：

```text
用户：请忘记我的饮品偏好。
```

状态迁移：

```text
preferred_drink.status = archived
preferred_drink.valid_to = request.valid_from or now
memory_events.operation = archive
memory_events.reason = user_requested_forget
```

归档后该槽位不再出现在 `list_active()`，因此不会进入普通 EvidenceBundle。如果目标不存在，只记录脱敏的 `forget_target_not_found` 拒绝事件。

### 4.3 已实现：检索时间衰减

普通检索只读取 `active`。每条 active 记忆再根据请求时间和 `valid_from` 计算时间分数：

```text
temporal_score = exp(-ln(2) * age_days / type_half_life_days)
```

较新的有效记忆获得更高时间分，较旧记忆按类型半衰期逐步降权。请求包含 `occurred_at` 时以请求时间为参考；否则使用系统当前时间。

### 4.4 已实现：禁存信息不进入状态表或审计原文

验证码、密码、私钥等信息会被拒绝：

```text
operation = reject
memory_id = null
不写 memory_state
memory_events 只写脱敏元数据，不写 candidate.value
```

示例：

```text
用户：顺便记住我的验证码 123456 和密码 apple-pass-9988。
```

结果：

```json
{
  "operation": "reject",
  "reason": "forbidden_sensitivity",
  "memory_id": null
}
```

`active_memories` 不增加。

拒绝事件示例：

```json
{
  "operation": "reject",
  "reason": "forbidden_sensitivity",
  "candidate": {
    "redacted": true,
    "type": "event",
    "predicate": "session_evidence",
    "scope": "actmem_session",
    "assertion_mode": "explicit",
    "sensitivity": "forbidden"
  }
}
```

### 4.5 已实现：按类型控制记忆寿命影响

| 类型 | 当前生命周期行为 |
|---|---|
| `fact` | 半衰期 730 天；新事实按时间门更新，旧事实保留为 historical |
| `relation` | 半衰期 365 天；接受同样的槽位与时间治理 |
| `summary` | 半衰期 365 天；参与统一 Top 10 和 active 检索 |
| `preference` | 半衰期 180 天；新偏好 supersede，未决偏好进入 tentative |
| `event` | 半衰期 90 天；同 Session 幂等强化 |
| `task` | 半衰期 30 天；任务按规范化槽位更新；Cross-Agent 架构演示统一为 `user_task / task:cross_agent_architecture_demo`，open 与 done 不会分裂成多个 predicate |
| `sensitive` | 半衰期 30 天；默认由隐私策略拒绝存储 |

系统不物理删除历史状态；通过 `active / tentative / superseded / archived` 明确区分当前可用值、待确认值、历史值和用户主动遗忘值。普通回答只使用 active。

## 5. 检索策略

检索目标不是召回所有相关文本，而是召回与当前意图有关、当前有效、可披露、可信的证据。

### 5.1 查询规划

`QueryPlanner` 是检索前置门控。当前采用“政策硬门 + 高精度规则 + LLM 三态判断 + 混合召回 + LLM 证据验证”的混合架构。核心目标是同时控制两类风险：不应使用记忆时发生越权或无效检索，以及检索到了语义相似但不能回答问题的个人信息。

#### 5.1.1 政策硬门

政策硬门先于规则和 LLM，后续模块不能覆盖：

| 条件 | 结果 | 示例 |
|---|---|---|
| 用户明确禁止使用记忆 | `NONE` | “不要使用长期记忆回答” |
| 当前 Session 上下文已经能解析“刚才/上面”等指代 | `NONE` | query 为“继续刚才的内容”，且 `recent_context` 非空 |

`SearchRequest.recent_context` 是可选的当前会话文本列表。它只用于区分当前 Session 与跨 Session 指代，并按 `memory_intent_context_max_chars=3000` 截断后提供给语义分类器。

#### 5.1.2 三态语义

判断不再只有 yes/no，而是：

| 状态 | 含义 | 后续行为 |
|---|---|---|
| `REQUIRED` | 没有历史记忆就无法可靠回答 | 直接执行完整检索 |
| `BENEFICIAL` | 记忆可能显著改善个性化，但不是回答必需条件 | 执行完整检索并接受证据验证 |
| `NONE` | 通用知识、当前输入处理或当前会话已足够 | 跳过数据库和 embedding |

为了兼容现有调用，`QueryPlan.needs_memory` 仍然保留：`REQUIRED/BENEFICIAL` 为 true，`NONE` 为 false；真实语义由 `QueryPlan.memory_intent` 表达。

#### 5.1.3 高精度规则评分

政策硬门未命中时，规则分析当前 query。不同信号对应记忆依赖分数，多个正向信号取最大值：

| 信号 | 分数 | 示例 |
|---|---:|---|
| 明确引用历史会话或记忆 | `0.98` | “我上次去日本住的是哪家酒店”“what did I tell you” |
| 跨会话省略或延续指代 | `0.90` | “继续上次的方案”“same as before” |
| 明确个人记忆对象 | `0.88` | “我的偏好”“我的任务”“where do I live” |
| 跨 Session 更新或纠正 | `0.86` | “现在改用 Python，不再用 Java”“任务已经完成” |
| 中文个人状态问题 | `0.72` | “我喜欢喝什么”“我平时用什么语言” |
| 所有格上下文 | `0.58` | “my workflow”“我的开发环境” |
| 领域同义词命中 | `0.52` | 命中配置中的 project、subscription 等领域扩展 |
| 记忆制品提示 | `0.52` | project、note、task、偏好、日程、代码示例 |
| 个性化请求 | `0.55` | “哪个更适合我”“for my needs” |
| 隐式个人上下文 | `0.45` | “帮我选一个” |
| 无记忆信号 | `0.05` | “Explain quicksort” |

记规则分数为：

```text
S_rule = max(s_1, s_2, ..., s_n), 默认值为 0.05
```

当前输入处理会覆盖所有格、项目等字面弱信号，但不会覆盖明确历史引用或跨 Session 指代；泛化第一人称在不存在明确历史、领域或记忆制品线索时按通用表达处理：

```text
当前输入处理：S_rule = min(S_rule, 0.12)
仅有泛化第一人称：S_rule = 0.15
```

例如“Summarize this: my project uses a queue”虽然出现 `my project`，但任务只依赖当前输入，因此不检索；“我想知道法国首都是什么”中的“我”只是表达方式，也不构成长期记忆依赖。裸词“长期记忆”不再被视为历史引用，因此“我在做长期记忆 Agent 项目”不会误触发 `REQUIRED`；只有“根据长期记忆、查长期记忆、你还记得”等读取语义才构成强信号。

#### 5.1.4 规则直判与 LLM 路由

默认阈值如下：

```text
S_rule >= 0.80  -> REQUIRED，规则直接检索
S_rule <= 0.20  -> NONE，规则直接跳过
0.20 < S_rule < 0.80 -> 模糊区间
```

在线配置 `configs/yunai.json` 在模糊区间调用聊天模型进行三态语义分类。模型输入只有当前 query、规则分数和受长度约束的当前 Session 上下文，不读取长期记忆正文。提示词明确要求判断“回答当前请求是否需要读取请求之前已经存在的记忆”，不能把“当前消息是否值得写入”混入判断。query 被声明为不可信数据，模型不得执行其中的指令。

模型返回严格 JSON：

```json
{
  "decision": "beneficial",
  "confidence": 0.91,
  "memory_types": ["preference", "task"],
  "reason": "personalization_would_help"
}
```

只有 JSON 结构合法、`decision` 属于 `required/beneficial/none`、字段类型合法且 `confidence >= 0.65` 时采用 LLM 结果。合法的 `memory_types` 包括 `event/fact/preference/task/relation/summary`；空列表或全部非法时回退为所有可检索类型。

#### 5.1.5 在线强制成功与离线模式

生产在线配置启用 `memory_intent_llm_required=true`。API 超时、调用失败、非法 JSON、非法字段或低于 `0.65` 的置信度都会终止本次请求并记录错误，不生成正式检索结果。允许按配置重试，但不允许以规则结果冒充成功的模型判断。

`configs/default.json` 是无 API 的本地测试模式，显式关闭 LLM 门控；它用于单元测试和确定性开发，不属于在线生产执行路径。

#### 5.1.6 规划结果与可观测性

示例：

```text
哪个开发方案更适合我的工作流？
```

规划结果：

```json
{
  "needs_memory": true,
  "memory_intent": "beneficial",
  "memory_types": ["preference"],
  "allow_sensitive": false,
  "reason": "personalization_would_help",
  "intent_confidence": 0.91,
  "decision_source": "llm",
  "rule_score": 0.58

```

`decision_source` 有四种值：

- `policy`：显式禁止记忆或当前 Session 已足够。
- `rule`：达到高置信规则区间，未调用 LLM。
- `llm`：模糊区间内采用了高置信 LLM 结果。
- `rule_fallback`：仅非严格的兼容配置可能出现；生产在线配置禁止该结果。

如果不需要长期记忆：

```json

  "needs_memory": false,
  "items": [],
  "trace": {
    "skipped": true,
    "reason": "generic_first_person",
```

### 5.5 词法分数：BM25 变体

词法分数使用 BM25 风格公式。

对每个查询词 `t`：

```text
idf(t) = log(1 + (N - df(t) + 0.5) / (df(t) + 0.5))
```

```text
bm25(t, d) =
  idf(t) * tf(t, d) * (k1 + 1)
  /
  (tf(t, d) + k1 * (1 - b + b * doc_len / avg_len))
```

参数：

```text
k1 = 1.4
b = 0.75
avg_len = 120.0
```

最终词法分数：

```text
lexical_score(q, d) = min(sum(bm25(t, d) for t in q_terms) / 8.0, 1.0)
```

### 5.6 Gemini Embedding 语义分数

在线配置使用 `gemini-embedding-2-preview`，通过现有 `YUNAI_API_KEY` 调用：

```text
https://yunai.chat/v1/embeddings
```

网关实际返回 3072 维向量。查询和文档采用非对称检索前缀：

```text
query:    task: search result | query: {query}
document: title: {predicate} | text: {record_text}
```

客户端对向量做 L2 归一化，语义分数为余弦相似度：

```text
embedding_score(q, d) = dot(normalize(E(q)), normalize(E(d)))
```

分数截断到 `[0, 1]`。

### 5.7 持久化向量缓存

同一个 SQLite 中维护 `memory_embeddings`：

```text
memory_id
model
dimensions
content_hash
vector_blob(float32)
updated_at
```

主键是 `memory_id + model + dimensions`。`content_hash` 由最终文档文本计算；文本变化时自动重新生成 embedding。已有记忆在首次搜索时惰性回填，新写入 active 记忆按配置立即建立索引。

Embedding 客户端具备超时、指数退避重试、并发上限、输入长度限制和维度校验。模型、维度或文本变化都不会错误复用旧向量。

### 5.8 Token Cosine 辅助信号与在线强制模型

系统保留 token 计数余弦作为独立信号：

```text
token_cosine(q, d) =
  dot(counts(q), counts(d))
  /
  (||counts(q)|| * ||counts(d)||)
```

当 Embedding 可用时，两种语义信号同时参与：

```text
online: 0.40 * embedding + 0.10 * token_cosine
```

生产在线配置启用 `embedding.required=true`。接口重试后仍失败、维度不符或返回空向量时直接终止，不输出 token-only 的正式结果。token cosine 是正常混合排序中的辅助通道，不是生产降级通道。缓存命中数、新建向量数和 API 状态写入 retrieval trace。

### 5.9 时间分数

检索首先执行有效时间硬门：

```text
valid_from <= request_time
and (valid_to is null or valid_to > request_time)
```

不满足条件的记录即使数据库状态仍为 `active`，也不会进入候选池、Embedding 或 EvidenceBundle。通过硬门的 Active 记忆再按请求时间和类型半衰期衰减：

```text
temporal_score = exp(-ln(2) * age_days / half_life_days)
```

请求提供 `occurred_at` 时以请求时间为参考，否则使用系统当前时间。

### 5.10 总分公式

当前重排总分：

```text
score =
  0.40 * embedding_score
  + 0.10 * token_cosine_score
  + 0.20 * lexical_score
  + 0.12 * temporal_score
  + 0.08 * importance
  + 0.07 * confidence
  + structured_type_bonus
  + predicted_type_match_bonus
  - privacy_penalty_if_sensitive
```

默认权重：

```json
{
  "embedding_weight": 0.40,
  "token_cosine_weight": 0.10,
  "lexical_weight": 0.20,
  "temporal_weight": 0.12,
  "importance_weight": 0.08,
  "confidence_weight": 0.07,
  "structured_type_bonus": 0.03,
  "memory_type_match_bonus": 0.04,
  "staleness_penalty": 0.30,
  "privacy_penalty": 0.40
}
```

结构化记忆加分：

```text
structured_type_bonus = 0.03 if memory_type != event else 0
```

类型预测加分只在 LLM 返回了类型子集时生效：

```text
predicted_type_match_bonus = 0.04
if predicted types are a strict subset and memory_type is predicted
else 0
```

敏感惩罚：

```text
privacy_penalty_if_sensitive = 0.40 if sensitivity in {"high", "sensitive"} else 0
```

最终分数会被截断到非负：

```text
score = round(max(score, 0.0), 6)
```

### 5.11 快速证据探测

系统在原始 query 的第一轮排序后、伪相关反馈前计算快速证据探测分，避免候选自身扩展出来的词制造虚假相关性。该分数用于 trace 和非在线模式的轻量门控；生产在线模式最终由 5.13 的 LLM 证据验证决定。

先对第一轮可披露候选计算：

```text
probe_score = max(
  lexical_score(d),
  token_cosine_score(d),
  embedding_score(d) if embedding available
  for every candidate d
)
```

非在线模式的放行条件：

```text
REQUIRED   -> 不应用探测门槛
BENEFICIAL -> probe_score >= 0.20
NONE       -> 不启动检索
```

探测结果写入：

```json
{
  "evidence_probe": {
    "applied": true,
    "top_score": 0.43,
    "threshold": 0.20,
    "accepted": true
  }
}
```

生产在线模式即使快速探测较低，也会让 LLM 证据验证器检查候选，避免固定阈值错杀同义表达。

### 5.12 伪反馈扩展

检索分两轮：

```text
base_terms = tokenize(query) + expanded_terms
initial = rank(records, base_terms)
feedback_terms = distinctive_terms(top initial docs)
final_terms = base_terms + feedback_terms
reranked = rank(records, final_terms)
```

默认：

```json
{
  "feedback_top_n": 3,
  "feedback_terms_per_doc": 6
}
```

`distinctive_terms` 的局部打分：

```text
term_score = count(term) * (1.0 + min(len(term), 12) / 12.0)
if term starts with digit:
    term_score += 0.5
```

### 5.13 LLM 证据验证、去重与证据包

重排后先执行披露过滤，再去重。原始事件按 Session 去重，结构化记忆按 `memory_id` 保留：

```text
event:      dedupe_key = source_session_id or memory_id
structured: dedupe_key = memory_id
```

这样同一轮中抽取出的“写文档喜欢安静”和“朋友聚会喜欢热闹”不会因为共享 `source_session_id` 而互相覆盖。

去重后最多取 20 个候选发送给 `LLMEvidenceVerifier`，避免多子问题请求中的低频槽位在验证前被 Top 10 截断。验证器只回答两个问题：候选是否足以支持当前请求，以及哪些 `memory_id` 是直接相关证据。语义相似、同属个人信息或排序分高都不能代替事实支持。

严格响应：

```json
{
  "sufficient": false,
  "confidence": 0.97,
  "relevant_memory_ids": [],
  "reason": "requested_hotel_fact_absent"
}
```

放行条件：

```text
sufficient == true
and confidence >= 0.70
and relevant_memory_ids 非空且全部来自候选集
```

多问题请求允许部分覆盖：只要至少一个子问题有直接证据，验证器就返回 `sufficient=true` 并选择可支持的候选，回答层必须明确哪些子问题缺少证据。单一事实问题仍要求该事实本身存在，因此无关个人记忆不能让“日本酒店”问题通过。

最终 EvidenceBundle 只保留验证器选中的候选，再按检索顺序取 `top_k`；因此验证候选池扩大不会改变最终 Top 10 上限。例如“我上次去日本住的是哪家酒店？”即使召回姓名、住址、饮品和旅行愿望，也会得到空 EvidenceBundle。验证 API 调用失败、返回非法 JSON、选择候选集外 ID 或低置信度时，生产配置直接失败，不使用未验证候选。

示例证据包：

```text
1. preferred_programming_language = python
2. preferred_drink = 茶
3. current_residence = 杭州滨江
4. session_evidence: 早期 Java 偏好
5. user_task: 周五前整理 Cross-Agent 架构演示材料
```

代码位置：

- `src/cross_agent/reader/memory_reader.py`
- `src/cross_agent/reader/scoring.py`
- `src/cross_agent/reader/evidence_verifier.py`

## 6. 使用策略

使用记忆时，Cross-Agent 不把整个记忆库交给模型，只给当前请求相关的 `EvidenceBundle`。
这样做的好处是大大降低上下文的token预算，实现agent越用越懂你且丝滑的效果。

### 6.1 LLM 提示词约束

当前系统提示词核心约束：

```text
Answer only from the provided EvidenceBundle and the user's current request.
If the evidence is insufficient or conflicting, say you do not have reliable memory evidence.
Do not invent personal facts.
Use a natural, concise answer; mention sources only when useful.
```

也就是说：

- 只能基于证据包回答。
- 不能编造用户事实。
- 证据不足或冲突时必须承认没有可靠证据。
- 回答要自然、简洁，不要机械罗列全部记忆。
- 只有在有必要解释历史变化时才提来源。

### 6.2 克制使用示例

问题：

```text
请根据长期记忆告诉我：我现在写接口示例应该用什么语言、默认喝什么、住在哪，以及还有什么任务？不要使用已经失效的信息。
```

合理回答：

```text
根据当前长期记忆：

- 写接口示例优先用 Python。
- 默认饮品是茶。
- 你住在杭州滨江。
- 还有一个任务：周五前整理 Cross-Agent 的架构演示材料，状态仍为 open。
```

不合理回答：

```text
你叫林澈，住在杭州滨江，喜欢咖啡，喜欢凌晨三点改需求，之前用 Java，现在用 Python，胃不舒服，所以我猜你最近压力很大。
```

问题在于：

- 使用了已失效的咖啡偏好。
- 把玩笑当作真实偏好。
- 加入了没有证据的推断。
- 过度披露了不必要记忆。

代码位置：`src/cross_agent/answer/generator.py`

## 7. 安全策略

安全分为写入安全和披露安全。

### 7.1 写入安全

写入前执行：

```text
PrivacyPolicy.can_store(candidate)
```

拒绝条件：

```text
candidate.sensitivity == "forbidden"
candidate.sensitivity in {"high", "sensitive"}
    and writer.allow_sensitive_storage == false
```

或命中敏感拒绝模式：

```json
[
  "password",
  "passcode",
  "verification code",
  "private key",
  "secret key",
  "credit card number",
  "social security number"
]
```

抽取器还会识别中文敏感词：

```text
密码
验证码
私钥
银行卡号
身份证号
```

示例：

```text
用户：顺便记住我的验证码 123456 和密码 apple-pass-9988。
```

结果：

```json
{
  "ingest_stats": {
    "extracted": 1,
    "created": 0,
    "reinforced": 0,
    "rejected": 1
  },
  "event_delta": [
    {
      "operation": "reject",
      "memory_id": null,
      "reason": "forbidden_sensitivity"
    }
  ]
}
```

关键点：凭证类信息不是“存了但不说”，而是**不进入可用长期状态**。

此外，硬敏感预扫描发生在 LLM 语义抽取之前。命中 forbidden 的原始会话不会发送给语义模型。`memory_events` 对所有 reject 操作默认启用：

```json
{
  "redact_rejected_event_payload": true
}
```

因此审计日志只保留拒绝原因、类型、predicate、scope、assertion_mode 和 sensitivity，不保留密码、验证码或原始 transcript。

### 7.2 披露安全

检索后还会执行：

```text
PrivacyPolicy.filter_disclosable(items, allow_sensitive)
```

规则：

```text
if sensitivity in {"high", "sensitive"}:
    disclose only if allow_sensitive is true
    and guard.allow_sensitive_disclosure is true
else:
    disclose
```

默认配置：

```json
{
  "allow_sensitive_disclosure": false
}
```

因此即使命中敏感记忆，也不会默认进入证据包。

### 7.3 用户隔离

所有存储查询都带：

```text
tenant_id
user_id
```

例如：

```sql
WHERE tenant_id=? AND user_id=? AND status=?
```

这保证不同租户、不同用户之间的记忆不会互相召回。

候选验证还要求：

```text
candidate.subject == candidate.user_id
```

LLM 输出中的租户、用户和来源 turn 不被直接信任，而是由系统从当前 Session 重新绑定。

代码位置：`src/cross_agent/policies/privacy.py`

## 8. 拒答策略

拒答的核心是证据约束，而不是泛泛地拒绝。

### 8.1 空证据拒答

如果检索结果为空，回答生成器直接返回：

```text
I do not have reliable memory evidence for that.
```

如果启用 LLM：

```text
EvidenceBundle 为空时不调用 API
```

这样避免模型在没有证据时自由发挥。

### 8.2 Guard 校验

`ResponseGuard` 规则：

```text
if require_evidence_for_personalization and not evidence.items:
    abstained = true
    answer = abstain_message
    used_memory_ids = []
else:
    abstained = false
    used_memory_ids = [item.memory.memory_id for item in evidence.items]
```

默认：

```json
{
  "require_evidence_for_personalization": true,
  "abstain_message": "I do not have reliable memory evidence for that."
}
```

### 8.3 拒答示例

问题：

```text
我大学在哪里读的？
```

如果长期记忆没有相关证据，应该回答：

```text
我没有可靠的长期记忆证据说明你大学在哪里读的。
```

不应该回答：

```text
你可能是在杭州读的，因为你现在住在杭州滨江。
```

后者是无证据推断。

代码位置：`src/cross_agent/guard/response_guard.py`

## 11. 策略总结

Cross-Agent 的长期记忆策略可以概括为：

```text
抽取：规则负责高精度与硬指令，LLM 负责开放语义，但都只生成候选。
解析：两路候选统一过滤、同值合并、写入评分、槽位消歧并取全局 Top 10。
准入：Schema、身份边界、置信度和隐私规则拥有最终否决权。
更新：先执行硬时间门，再按值、分数和断言方式判断 reinforce / supersede / historical / tentative / promote。
遗忘：通过 superseded、用户 archive、valid_to 和 active-only 检索实现。
检索：先按请求时间硬过滤 active 记忆，再执行 Gemini Embedding + BM25 + token cosine + 时间衰减混合重排，最后经 LLM 证据验证筛成 EvidenceBundle；在线模型失败不生成正式结果。
使用：只把 EvidenceBundle 交给回答器，避免全量记忆污染上下文。
安全：LLM 前预扫描、写入前禁存、reject 脱敏、披露前过滤、tenant/user 隔离。
拒答：没有可靠证据时承认不知道，不编造个人事实。
```

最终目标是让长期记忆成为可审计、可更新、可撤回、可约束的证据系统，而不是一个不断堆积、随意调用的聊天记录仓库。
